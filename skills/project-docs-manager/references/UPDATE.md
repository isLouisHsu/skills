# UPDATE — 项目文档迭代更新

← 返回 [SKILL.md](../SKILL.md)

## Purpose

引导 AI 完成一次完整的文档驱动迭代：读取现状 → 收集输入 → 分析 → 提出方案 → 实施 → 记录效果 → 更新文档。

## Update Flow

### Step 1: 读取文档库现状

```
1. 读取 {docs_path}/_INDEX.md → 了解文档库结构
2. 读取 {media_path}/OVERVIEW.md → 了解项目当前状态、核心指标、待办事项
3. 读取 {media_path}/CHANGELOG.md → 了解最近的变更历史（重点看最近 5 条）
4. 如有需要，读取相关的 {media_path}/changes/*.md 或 {media_path}/knowledge/*.md 深入了解
```

AI 在此步骤形成对项目现状的完整理解，无需用户重复讲解背景。

### Step 2: 收集本次迭代输入

向用户收集以下信息：

| 字段 | 必填 | 说明 |
|------|------|------|
| **背景** | 是 | 为什么要做这次变更？触发原因是什么？ |
| **优化方向** | 是 | 具体要改什么？目标是什么？ |
| **预期效果** | 否 | 期望达到什么结果？有没有量化指标？ |
| **约束条件** | 否 | 有什么限制？（时间、资源、兼容性等） |
| **关联知识** | 否 | 是否涉及需要沉淀的新知识？ |

如果用户在触发时已经提供了部分信息，直接使用，只补问缺失的必填字段。

### Fast Path: 快速记录模式

如果用户**已经提供了完整的变更内容**（背景 + 做了什么 + 结果），且明确只想**记录**这次变更而非重新分析方案，则：

- **跳过 Step 3（分析与方案）**
- **跳过 Step 4（实施）**
- **直接进入 Step 6（更新文档库）**，将用户提供的内容整理后写入变更记录

> 示例触发语："帮我记录一下，今天我把登录接口改成了 JWT 认证，背景是… 结果是…"

### Step 3: 分析与方案

基于文档库现状和用户输入，AI 执行：

1. **差距分析**：当前状态 vs 目标状态，明确要解决的问题
2. **方案设计**：提出 1-3 个可行方案，说明优劣
3. **推荐方案**：给出推荐并说明理由
4. **影响评估**：列出受影响的文件、模块、指标

输出格式：
```markdown
## 分析结果

**当前状态**：（一句话）
**目标状态**：（一句话）
**差距**：（要点列表）

## 方案

### 方案 A：{名称}（推荐）
- 做法：...
- 优势：...
- 风险：...

### 方案 B：{名称}
- 做法：...
- 优势：...
- 风险：...

## 影响范围
- 涉及文件：...
- 涉及指标：...
```

等待用户确认方案后再继续。

### Step 4: 实施

按照确认的方案执行变更。实施过程中：

- 遵循项目已有的编码规范和约定
- 每完成一个模块，请求用户评审（遵循编码规则 #7）
- 记录实施过程中的关键决策和发现

### Step 5: 记录效果

实施完成后，收集效果数据：

- 如有量化指标，记录变更前后的对比
- 如无量化指标，记录定性评估
- 记录遗留问题和后续 TODO

### Step 6: 更新文档库

**必须更新**的文档（按顺序执行）：

#### 6.1 创建变更详情文件

文件路径：`{media_path}/changes/{YYYY-MM-DD}_{简短标题}.md`

模板 → [templates/change_detail.md](./templates/change_detail.md)

#### 6.2 更新 {media_path}/CHANGELOG.md

在变更记录表格最上方（时间倒排）追加一行：

```markdown
| {YYYY-MM-DD} | {变更标题} | {一句话结果} | [详情](./changes/{文件名}.md) |
```

#### 6.3 更新 {media_path}/OVERVIEW.md

更新以下 section：
- **当前进展 > 状态**：反映最新状态
- **当前进展 > 最近里程碑**：追加本次变更
- **当前进展 > 待办事项**：根据本次变更的 TODO 更新
- **核心指标**：如有指标变化，更新数值

#### 6.4 更新 {docs_path}/_INDEX.md

- 如有新文件，追加到对应 section 的表格中
- 更新相关文件的"最后更新"日期

#### 6.5 创建/更新知识文件（如需要）

如果本次迭代产生了值得沉淀的知识：

1. 创建 `{media_path}/knowledge/{topic_slug}.md`，模板 → [templates/knowledge_detail.md](./templates/knowledge_detail.md)

2. 在 `{media_path}/KNOWLEDGE.md` 索引表格中追加条目
3. 在 `{docs_path}/_INDEX.md` 的知识详情 section 中追加条目

### Step 7: 验证与报告

完成所有文档更新后：

1. **链接验证**：检查所有新增的交叉引用链接是否指向正确的文件
2. **索引一致性**：确认 `{docs_path}/_INDEX.md` 与实际文件列表一致
3. **输出迭代报告**：

```markdown
## 迭代完成报告

**变更**：{标题}
**结果**：{一句话}
**更新的文档**：
- {文件1}：{更新内容}
- {文件2}：{更新内容}

**下一步建议**：
- {建议1}
- {建议2}

**遗留 TODO**：
- [ ] {TODO1}
- [ ] {TODO2}
```

## AI Self-Driven Iteration（AI 自驱迭代）

当用户要求 AI "自主分析并提出优化方向"时，AI 应：

1. 完整读取文档库（`{docs_path}/_INDEX.md` → `{media_path}/OVERVIEW.md` → `{media_path}/CHANGELOG.md` → 最近变更详情）
2. 识别以下信号：
   - `{media_path}/CHANGELOG.md` 中未关闭的 TODO
   - `{media_path}/OVERVIEW.md` 中待办事项
   - 变更详情中的遗留问题
   - 指标趋势中的异常
3. 形成优化建议列表，每条包含：
   - 优化方向
   - 预期收益
   - 优先级（高/中/低）
   - 依据（引用文档中的具体数据或记录）
4. 提交给用户确认后，进入 Step 3 开始正式迭代

## Error Handling

| 场景 | 处理 |
|------|------|
| 文档库未初始化 | 提示用户先运行初始化流程 → [INITIALIZE.md](./INITIALIZE.md) |
| 索引与实际文件不一致 | 先修复索引，再继续更新流程 |
| 同一天多次变更 | 文件名加序号：`YYYY-MM-DD_title_2.md` |
| 变更详情过长 | 拆分为多个子文件，在主变更文件中用链接引用 |
| 用户未提供预期效果 | 跳过量化指标，记录定性评估，在 TODO 中标注"待补充效果数据" |
